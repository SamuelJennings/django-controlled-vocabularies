# Brainstorm: django-controlled-vocabularies

Working design notes from the idea and viability grilling that preceded this repository. These are
the current leanings and the reasoning behind them, not ratified decisions. Real architectural
decisions are made per feature, during that feature's own grilling. Treat this as background
thinking that a feature can lean on or overturn, not a contract.

## Scope: SKOS only

Controlled vocabularies in research infrastructure span SKOS, RDF, and OWL. The predecessor projects
(`django-research-vocabs`, then a general rdflib builder) tried to stay format-general and became
diffuse: the API lost focus and the design never settled. In practice the vocabularies that need
managing are SKOS concept schemes.

So the lean is to target SKOS only. The models, editor, import, and export are shaped around SKOS
concept schemes, concepts, collections, labels, and semantic relations, not arbitrary RDF or OWL,
and with no reasoning or inference. Non-SKOS predicates met on import are preserved verbatim as
escrow (see the storage note) rather than modelled: they round-trip, they are not understood. OWL
and general RDF stay rdflib's job.

## Storage: relational models are the source of truth; RDF is a projection

A vocabulary manager can hold its data as serialized RDF files, in a triplestore, or in relational
models. The predecessors kept vocabularies as RDF (in git, or in memory via rdflib) and kept hitting
the same walls: querying needed deep RDF knowledge, large vocabularies (tens of thousands of
concepts, such as NASA GCMD) were awkward to load and search, and there was nowhere natural for
editing, permissions, or referential integrity.

The lean is to store vocabularies as Django models in a relational database, which are the single
source of truth, and to treat RDF as a projection generated by rdflib only at the import and export
boundary. rdflib is not used for querying or in-memory representation. That puts everything Django is
good at (forms, admin, CRUD, object permissions, indexed search, autocomplete, foreign-key
integrity, migrations) directly to work, and large vocabularies become ordinary indexed rows.
Published RDF is served by Django at stable URIs. The system is not a triplestore and exposes no
SPARQL endpoint.

## Concept storage: a document plus projected index columns

The SKOS schema defines many predicates, but published vocabularies fill only a small core
(prefLabel, definition, hierarchy) and leave the long tail empty. Thirty sparse nullable columns buy
migration weight and little else, and hand-mixed schemas in the wild (for example
`dcterms:description` where `skos:definition` was meant) do not fit fixed columns cleanly.

The lean is to make a concept's editable state a JSON document of predicate-to-object pairs, the
sole source of truth for its literal-valued and unknown predicates, and to project a few hot
predicates (`uri`, `pref_label`, `definition`) into concrete, read-only, indexed columns synced from
the document on save. The document holds everything; the columns are a materialised index derived
from it, so there is one editable representation and the columns cannot drift. Relations do not go in
the document (they are their own links, see below). Import can normalise via a configurable
predicate-alias map, surfaced to the user; unmapped predicates fall through to the document
untouched, which is the lossless escrow for managed vocabularies. Writes have to go through the sync
path, since bulk operations bypass `save()`.

## Translations: a shared and translated split

SKOS labels and notes are language-tagged and often multi-valued per language (many `altLabel`s per
language). Per-object translation frameworks assume single-valued fields per language, which fights
SKOS. But some predicates (`notation`, mappings, URI-valued and typed literals) are
language-independent and must not be duplicated per language.

The lean is django-parler with the concept split into two documents: a shared document on the base
model for the language-independent predicates, and a translated document inside parler's translated
fields for the language-tagged ones. A literal with a language tag routes to the translated document;
everything else routes to the shared one. Multi-valued labels are lists inside the translated
document, so SKOS's many-per-language labels fit without straining parler. The app's language set is
deploy-time config: importing a language-rich external vocabulary keeps only the app's supported
languages and drops the rest (a deliberate normalisation on external import, surfaced to the user),
and a later re-import after adding a language fills in the newly supported one. Managed vocabularies
round-trip losslessly within the app's configured languages.

## Relations: a canonical-direction self-referencing link

SKOS concepts form a graph: `broader` and `narrower` (an inverse pair), `related` (symmetric), plus
cross-vocabulary mappings. SKOS permits polyhierarchy (a concept may have several broader concepts),
which single-parent tree libraries cannot represent.

The lean is a self-referencing many-to-many on the concept with a through model carrying a
relationship type. Store one canonical direction and derive the inverse (persist `broader`, read
`narrower` back), so the data can never assert one direction without the other. Normalise the
symmetric `related` to one row. Mappings are not local relations: they usually point at URIs with no
local row, so they live in the document (or a dedicated model later if they need to be queried).
Subtree queries need recursive CTEs rather than a tree-library call, which is a day-one need for
subtree-scoped autocomplete.

## Identity and lifecycle: the URI is identity

Once downstream user data holds references to concepts, those concepts stop being disposable. A
re-import that deletes and recreates concepts breaks every referencing row, and a curator deleting a
concept that thousands of records reference destroys data.

The lean: identity is the concept URI, never the database primary key, and the URI is immutable after
first publish (labels can be renamed freely, the URI never moves). Import upserts by URI rather than
delete-and-recreate. Concepts have a status lifecycle (`draft`, `published`, `deprecated`, emitted as
`owl:deprecated`), so a referenced concept is retired by deprecation, not deletion, and references
use `on_delete=PROTECT`. The editor has to make deprecation prominent, or curators will feel the
protection as an obstacle. These identity and upsert invariants are the load-bearing ones.

## The predicate registry

The editor is a key-to-value document editor: rows of predicate and object, added as needed, with the
predicate chosen from a dropdown. A flat string-to-string widget cannot express what vocabulary data
needs, because values have RDF types (a definition is a language-tagged literal, an `exactMatch` is a
URI, a `notation` is a typed literal, an `altLabel` is a list) and predicates have cardinality (one
prefLabel per language, many altLabels).

The lean is a predicate registry: a curated map of CURIE to its value type, cardinality, and whether
it is translatable. It drives the dropdown, the per-row widget, validation, the shared-versus-
translated routing, and the import alias normalisation, all from one place. Unregistered predicates
still work as free-text escrow, so arbitrary vocabularies keep importing losslessly; the registry
just makes the known ones safe and pleasant. Export then emits the correct URI, literal, or typed
literal because the type is declared rather than guessed.

## Consumption: a relationship field, not choice lists

The original motivation was integrating vocabularies into Django as choice fields. But a `choices=`
iterable bakes vocabulary content into schema and code: adding a concept means a code release of
every consuming app, there is no referential integrity, and a choice tuple cannot carry a concept's
definition or URI.

The lean is to treat consumption as a relationship: a field wrapping a foreign key to a concept and
one wrapping a many-to-many, each constrained to a scheme and rendered with autocomplete. Vocabulary
evolution then becomes a data operation (a curator adds a concept and every consuming form picks it
up, no deploy), referential integrity and the concept's definition and URI come for free, and large
vocabularies work as autocomplete-backed lookups rather than in-memory lists.
